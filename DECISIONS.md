# Decision Log

A running record of points where my judgment redirected the AI's default. The split in [WORKFLOW.md](WORKFLOW.md) says I direct and the AI implements — this is the evidence for that claim, kept deliberately so it shows the full range: roadmap-level strategy down to one-line craft fixes. Nothing here is the AI's idea that I rubber-stamped; each is a call I made against what the AI was about to do.

Entries are factual — what the AI defaulted to, what I decided instead, and why. The reasoning is meant to carry the weight, not adjectives.

## Strategic and architectural calls

| Date | The AI's default | My call | Why |
|---|---|---|---|
| 2026-05-27 | Build breadth-first — every problem type first, turn the agent on around Week 7. | Re-sequence to **agent-first**: the agentic loop is the v0.1 milestone; everything after is capability expansion. | Building breadth before the agent buries the differentiator. Sequence toward the thing that matters first. |
| 2026-05-28 | Leave sandboxed code-gen ("the agent writes + runs any training code") late in the roadmap. | Bump it **early to v0.2**, right after the v0.1 loop. | "Run the model the research actually recommends" is the real unlock — it shouldn't wait behind six other versions. |
| 2026-05-27 | Frame cost-constrained optimization as *the* moat. | Reframe as the **specialized combination**; cost-aware serving is the *flagship*, not the only moat. | One feature is not defensible. The defensibility is the combination plus the domain specialization. |
| 2026-05-28 | Candidate spec as a flat `{"model": x, ...hyperparams}`. | Nested `{"model": x, "params": {...}}`. | Clean separation of *which model* from *its params*, no key collisions, and the exact shape the agent will emit from research. |
| 2026-05-26 | Treat a reported/prior baseline score as the number to beat. | **Never trust a reported score** — always re-measure the baseline through our own eval. | A number computed a different way is not comparable; you cannot honestly claim an improvement against it. |
| 2026-05-29 | Take `--baseline` + `--source` as context only, no behavior; defer reproduction to Week 10. | **Make the inputs do something in v0.1** — by end of Week 3, source must reconstruct the baseline approach through our eval and then improve from there. | A baseline number we never use is dead weight. If you take an input, it has to drive something. |
| 2026-05-29 | Implied user-provided source code would be executed in the v0.2 sandbox to reproduce a baseline. | **Never execute user-provided source code — ever.** Source is read-only text the LLM uses to reconstruct the approach; the v0.2 sandbox runs the agent's OWN generated code, never the user's. | User notebooks/scripts are untrusted (malware/RCE-adjacent); a sandbox reduces but doesn't eliminate the risk. Reconstruct-from-text is the right permanent policy. |
| 2026-05-30 | Defer all interactive-CLI features (pause, mid-run chat, streaming, Ctrl-C) to the Week-12 Streamlit UI. | **Split into two milestones.** v0.2 picks up the *cheap* interactive wins (live progress display, streaming LLM responses, graceful Ctrl-C) alongside sandboxed code-gen; v0.3 is a new focused milestone for *full mid-run chat* (pause/resume/conversational state). Streamlit UI shifts to v0.10. | Waiting until Week 12 for any interactivity is too late; lumping it all into v0.2 balloons that milestone. Two focused milestones ship each piece when it's actually ready. |
| 2026-06-01 | Keep it single-agent (one Proposer LLM in a deterministic loop) — multi-agent adds coordination cost for little gain. | **Go multi-agent after v0.2**, at the Researcher milestone (v0.4): specialist agents (Researcher, Proposer, Critic) report structured, typed output to a **supervisor agent** that decides. v0.1/v0.2 stay single-agent; built on our own harness, not LangGraph. | Specialists with one focused job tool-call far more reliably (mitigates the local-model yield problem), and a supervisor that reasons over digested high-quality context makes better decisions than one LLM juggling everything. The Researcher is the natural single-to-multi transition. |
| 2026-06-02 | On the code-gen path, bound the agent to a library list (either an allow-list or one auto-derived from the installed environment, fed into the prompt). | **No library boundary at all** — the agent imports whatever it wants and we install its imports before running. The environment adapts to the code, not the code to the environment. Resolution is deterministic (AST extraction, not a second LLM); auto-install happens only in the disposable sandbox, never the user's local venv. | An allow-list is the wrong frame for "the agent writes its own code" — it re-imposes the v0.1 cap we're lifting. A bad/exotic import just fails its install and becomes a captured failure + retry, so openness costs at most one iteration. The import→package **resolution architecture is provisional/TBD** (hand-kept alias map for now); to be revisited. |
| 2026-06-03 | Ship sandboxed code-gen behind a flag (`--code`), e2b as the safe default compute, and don't install into the user's local venv. | **Code-gen is the DEFAULT** mode and **local is the DEFAULT** compute; local install-on-demand is allowed **with consent**. A setup wizard saves these once so users don't retype flags. | Code-gen is the v0.2 headline — hiding it behind a flag buries it. Local-default is lower friction for a developer tool (running generated code is normal, like running any script) and is made a conscious choice by the first-run wizard. Consent (asked once) is what makes local install safe, which is what makes "no library boundary" actually true on the default path. |
| 2026-06-03 | Reconstruct a `--source` baseline as an allow-listed `{model, params}` spec, approximating anything off-list to the "closest equivalent." | **Reconstruct as CODE** — the Reconstructor writes a `train_and_predict` reproducing the source faithfully (real CatBoost, a custom net), no approximation. | Now that the agent runs its own code, forcing the user's existing approach through an allow-list throws away fidelity for no reason. The baseline should be *their* approach, measured through our eval — not our nearest guess at it. |
| 2026-06-03 | Feed only the final score (or error reason) back into the next proposal. | **Feed the run's stdout + error traceback back** to the code proposer (bounded, recent iterations). | If the agent can see what its code printed (class balance, dtypes, missing values) and the real traceback when it crashed, it learns the data and self-corrects — turning each run into exploration, not just a blind score. Leakage-safe because holdout labels never enter the script. |
| 2026-06-04 | History shows the proposer only a one-line description + score of past attempts. | Also feed a **deterministic component fingerprint** (`[used: SimpleImputer, OneHotEncoder, …]`, AST-extracted, no LLM) per attempt, and make the prompt **feature-engineering-first**. | Live runs showed the agent repeated the same impute+one-hot and only swapped the model, because it literally couldn't see prior preprocessing. The fingerprint makes "what was tried" visible; an A/B proved the FE staleness was prompt-bound (even a 70B only swapped models) — and the FE prompt then made local qwen engineer a new feature and beat every prior run. The richer LLM-summary version is deferred to v0.4. |
| 2026-06-04 | Build sandboxed code-gen as one-shot (write a whole `train_and_predict`, run once); defer interactive cell-by-cell to a later milestone (v0.3). | **Pull cell-by-cell execution into v0.2**: a stateful code-interpreter session where the LLM writes a cell, sees its real output, then writes the next — within one experiment. | One-shot is blind: the agent writes the whole pipeline without ever looking at the data, so it reflexively does generic preprocessing and, when pushed to engineer features, silently breaks the pipeline (near-zero scores it can't see coming). Looking-at-the-data-as-you-build is the capability unlock, and it's worth doing now rather than shipping a blind v0.2. |
| 2026-06-05 | Keep the multi-agent split for v0.4 (the Researcher milestone); v0.2 stays single-agent. | **Re-sequence: the Supervisor + Coding agent land IN v0.2** (coder-first build). The supervisor compresses runs 1..n-1 into a 1-2 line brief for run n's coder; specialists (Researcher/Critic) still graduate later at the same tool boundary. | Cell-by-cell already needs an across-experiments strategist to brief each session and decide stop — that IS the supervisor. Building it now (one LLM via tools) is the honest start of the multi-agent system, not a deferral; the tool boundary is where the future specialist agents plug in unchanged. |
| 2026-06-05 | Cap the cell-by-cell session by a max number of cells. | **No cell cap — the wall-clock deadline is the bound**, and the deadline charges KERNEL-execution seconds only, never LLM thinking time. (`max_cells` stays as a runaway backstop, not a budget.) | A cell count is the wrong unit: small inspection cells are cheap and good, and charging the local model's slow thinking against the budget would starve a weak driver of turns a fast one gets for free. Bounding by kernel-time gives any backend the same working budget. |
| 2026-06-06 | Let the agent treat `X_train`/`X_holdout` as ordinary mutable working data. | **Protect the canonical inputs in the harness** — snapshot them in the preamble and restore before every agent cell, so in-place mutation in one cell cannot silently poison later attempts in the session. | A weak model reflexively mutates `X_train` in place (`X_train[cols] = enc.fit_transform(...)`); across cells that corrupts the data every later attempt builds on. Immutability is a guarantee the model shouldn't have to think about — enforce it structurally, like verified-finish. |
| 2026-06-07 | Keep iterating the coder prompt in-session to force stepwise (cell-by-cell) code and push the f1 quality bar. | **Stop the in-session prompt loop; author the coder prompt out-of-band** to reach the quality bar, and reserve this session for the day-by-day build. Staged-vs-monolithic writing style is **model-bound, not harness-bound**. | Proven by runs: a 14B defaults to a complete script and reverts to monolithic cells whenever handed a working blob to edit; prompt wording can't reliably beat that, and forcing it regressed reliability (0.5813, 2/5) vs the monolithic harness (0.6353, 5/5). The harness lifts the floor model on score + reliability; it doesn't need to make a weak model *write* like a strong one. Stepwise R&D presentation, if wanted, belongs at the deliverable layer. |
| 2026-06-08 | Treat qwen3:14b as the fixed floor model and keep working the prompt/harness around it. | **Change the floor model — gemma4:12b deserved a chance** (and test thinking mode empirically too, rather than assuming it's not worth the latency). | The floor model is a choice, not a constant. Validated immediately: the unchanged harness ran 10/10 experiments in the 0.60–0.62 band on gemma — the first all-iterations-above-baseline run. The thinking half came back refuted (it made cells MORE monolithic and broke the supervisor's tool calls — now coder-only, off by default), but refuted-by-run beats assumed-by-opinion, and the experiment left two durable instruments: the coder-only think split and the thinking-trace-in-notebook debugger. |
| 2026-06-08 | Presented the staging-vs-score result as a tradeoff to weigh (staged R&D notebooks at 0.59 vs the unreadable monolith's 0.6353), with reverting to the monolith on the table. | **R&D staging is LOCKED; the score delta is acceptable.** Proper research-style notebooks — explore, decide, build in steps, conclude — are the product bar, not just the metric. | "If the code being written is in proper R&D style that's better." The notebook IS the deliverable a human reads; a number without a readable journey isn't the product. Also overturned the day-old "staging is model-bound" conclusion: it was model-bound for prose RULES; a worked example the model imitates fixed it (monolithic cells 31–35% → <1%). The score gap closed two days later anyway (0.6325 staged). |
| 2026-06-08 | Keep the Summarizer agent at v0.4 as planned; the supervisor's one-line history compression is enough for v0.2. | **Pull the Summarizer into v0.2 now**: one digest per finished notebook (what was tried, data insights, what helped/hurt, takeaway), stored in Memory, read by the supervisor — never whole notebooks as context. | The runs hit the exact wall the Summarizer exists for: winning techniques (TargetEncoder, threshold tuning) were discovered in one notebook and DROPPED by the next, because only a 1-line brief crossed between them. And whole-notebook context would hallucinate by iteration 5 — compression is the only scalable carrier. The digest is the unit of cross-notebook knowledge transfer. |
| 2026-06-09 | Land the prompt rewrites by committing and merging to main once tests pass. | **Open a PR and do NOT merge until a run satisfies us; every prompt rewrite isolated in its own commit.** `main` stays the known-good earlier-prompt baseline. | Prompts are the piece we're least sure of and most likely to want back: a one-commit `git revert` must drop a prompt alone and keep the infrastructure. Paid for itself the same week — the validate-never-submit regression and the supervisor rewrite both iterated on the branch while `main` stayed comparable. |
| 2026-06-10 | Treat the it2 missing-column loop as another error-recovery bug to patch around. | **State the design principle: open workspace, guarded boundary.** Restrictions exist ONLY at the validation boundary (sealed holdout labels, verified predictions contract, pristine canonical input frames); everything inside the workspace — any engineered feature, library, model — is open and persists in the model's own variables. Fix the guard's VISIBILITY, never loosen it. | "We need to give the model the ability to do feature engineering — that's what R&D is. Allow everything, but restrict some things so we can validate every run properly." The incident wasn't an FE restriction — the model's in-place patch was silently undone by the input-reset guard it couldn't see. The fix arms the model with the reset fact at the exact moment it loops (the same-error nudge), keeping both the open solution space and the validation guarantee. |
| 2026-06-13 | Ship all three "cheap interactive wins" the 2026-05-30 split promised for v0.2 — including token-level streaming of LLM responses. | **Ship graceful Ctrl-C and per-cell progress logging in v0.2; defer token-level LLM streaming to v0.3.** v0.2 closes with the run-loop interrupt handler (finalize Memory + keep earned notebooks, no stack trace) and a per-cell progress line (which cell, ok/error, kernel-time budget spent). | The cell-by-cell loop already solved liveness structurally — the unit the user watches is the committed cell and its real output, not the tokens en route to it, and a per-cell line makes a long local-model session visibly move. Streaming partial tokens through a tool-calling loop (parsing incomplete tool-call JSON, interleaving thinking traces, doing it across both the OpenAI and native-Ollama adapters) is real client work for a marginal payoff when cells are the unit. It belongs in v0.3 next to the mid-run chat that genuinely needs token-by-token output, not bolted onto v0.2 to honor a checklist. |

## Craft and detail calls

| Date | The AI's default | My call | Why |
|---|---|---|---|
| 2026-05-27 | Treat slow model tests as "probably my laptop," or fix on a hunch. | Fix **only if measurement proves** it's a real bug, not the hardware. | Don't optimize on vibes — and a 10s fit on 120 rows is never the hardware. Measuring found a ~200x thread-oversubscription bug. |
| 2026-05-25 | Use emoji status markers in the public docs. | Drop decorative emoji; words for status, plain `✓`/`✗` in matrices. | Emoji-as-status reads "AI-generated / beginner" in serious infrastructure repos. |
| 2026-05-27 | Leave the BUILD_LOG inconsistent (Week 1 had a "Done?" column; Weeks 2–3 didn't). | Add the column everywhere. | Asymmetry in the public trail looks careless. |
| 2026-05-26 | Date a research entry to the day it was written up. | Correct it to the day the research actually happened. | The decision trail is only useful if it's accurate. |
| 2026-05-27 | Mixed day-numbering in the session notes (plan-position vs calendar). | Reconcile to calendar-day headers with plan-position in parentheses. | One unambiguous timeline. |
| 2026-05-30 | When `/v1` can't disable qwen3 thinking, bundle the workaround into `OpenAICompatibleClient`. | Build a separate **`OllamaClient`** adapter on Ollama's native `/api/chat` with `think:false`; the OpenAI client stays clean for cloud backends. | One backend's quirk shouldn't pollute the shared client. The `LLMClient` protocol is the seam — adding a sibling adapter is the right shape. |
| 2026-05-30 | Hardcode prompt strings inside `core/proposer.py`. | Centralize every prompt in `src/iterate/prompts/prompts.yaml` (a loader exposes `PROMPTS`); modules reference structured keys. | Prompts change far more often than logic; one file = one place to review and iterate wording. |
| 2026-05-30 | Accept `--baseline X` alone (informational-only number; we'd still use the factory default for actual measurement). | **`--baseline` requires `--source`.** A number with no source describes nothing we can run, so the input is dead weight on its own. | Either it drives behavior or it shouldn't be a flag. Explicit CLI error is better than silent ignoring. |
| 2026-05-30 | `--fresh` should delete the existing memory db (true reset). | Archive it instead — rename to `<name>.YYYYMMDD-HHMMSS.bak`, then create a new clean db. Triggered by `--fresh`, `--source`, or `--baseline + --source` — any explicit "new chapter" signal. | Recoverable cheap safety net; honors the "start over" intent without losing forever. The user can `rm` the `.bak` if they really want it gone. |
| 2026-05-31 | Output the best model as just a config + score in the summary. | **Save the winning model as a usable `joblib` artifact** (+ a `best.json` config sidecar), printed with a load line. | "We found the best model" is hollow if the user has to rebuild it themselves. The contract promised an *artifact* — deliver one they can `joblib.load(...).predict(X)`. |
| 2026-05-31 | Ship every dependency (incl. forward-looking ones) in the core install. | **Lean core deps** — drop the unused `sqlalchemy`; move `e2b` (sandbox/v0.2) and `kaggle`/`datasets` (future adapters) to extras. | `pip install iterate-ai` should pull only what v0.1 actually runs on; speculative deps belong behind extras. |
| 2026-06-03 | Keep e2b behind a `[sandbox]` extra (`pip install 'iterate-ai[sandbox]'`) per the lean-core rule. | **Bundle e2b into core** so the only command is `pip install iterate-ai`; `--compute e2b` then needs just an API key. (Supersedes the e2b half of the lean-core call above; kaggle/datasets/etc. stay extras.) | Users don't optimize install commands — they expect one boring line and accept it taking time. Bracket-extra syntax is friction for no payoff when the dep (e2b) is now a v0.2 default-path backend and is a light HTTP SDK. Lean-core still holds for genuinely optional/heavy deps. |
| 2026-06-04 | A cloud `--backend` alias (groq/openai/…) needs the user to also pass `--base-url`, or it falls back to the local Ollama URL. | **The alias supplies its own base URL** (`groq`→`api.groq.com/openai/v1`, + openai/together/deepseek); explicit `--base-url` still overrides. | Surfaced by a saved-config run that 404'd: storing `backend=groq` without a URL was useless. The alias already names the provider, so it should imply the endpoint — which is what makes "the setup wizard need only provider+model+key" actually work. |
